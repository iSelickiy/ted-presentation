# Промпт для агента: собрать рабочую папку routine-agent

Этот файл можно скопировать в Codex или Claude Code. Он нужен, чтобы агент взял ваш заполненный бриф и собрал рабочую папку с инструкциями, локальной базой и шаблонами.

Перед запуском подготовьте короткий бриф по инструкции из `02-human-setup-guide.md`: название рутины, цель, источники, правила безопасности, нужные действия, правила дублей и формат результата.

---

```text
Ты помогаешь мне собрать рабочую папку для личного routine-agent.

Цель: создать переносимую локальную систему, которая помогает регулярно собирать контекст из моих рабочих источников, готовить follow-up, CRM notes, задачи и handoff, но ничего не записывает во внешние системы без моего отдельного явного согласия.

Сначала прочитай мой бриф ниже. Затем создай рабочую папку и файлы инструкций. Не подключайся к внешним системам и не выполняй реальные записи на этом этапе. Нужно только подготовить локальные файлы и тестовый dry-run сценарий.

## Мой бриф

[ВСТАВЬТЕ СЮДА ЗАПОЛНЕННЫЙ БРИФ:
- название рутины;
- цель;
- период по умолчанию;
- источники;
- что считать важным апдейтом;
- какие действия готовить;
- что нельзя делать без согласия;
- что считать дублем;
- материалы и ссылки;
- желаемый формат результата.]

## Что нужно создать

Создай папку:

routine-agent/

Внутри создай структуру:

routine-agent/
  START_HERE.md
  operating-context.md
  routine-flow.md
  data/
    clients.yaml
    projects.yaml
    materials.yaml
    action-log.yaml
  templates/
    review.html
  runs/
    .gitkeep

Если моя рутина не про клиентов, замени `clients.yaml` на более подходящее имя, например `entities.yaml`, `candidates.yaml`, `accounts.yaml` или `projects-index.yaml`. Выбери одно название и используй его последовательно во всех инструкциях.

## Требования к START_HERE.md

Сделай этот файл стартовой точкой для нового чата с агентом.

Включи:

- короткое назначение рутины;
- список файлов, которые агент должен читать перед работой;
- стандартный запрос для запуска регулярного прогона;
- правило периода по умолчанию;
- список обязательных источников;
- правило остановки при недоступном источнике;
- главное правило безопасности: ничего не писать наружу без согласия;
- ссылку на `operating-context.md` и `routine-flow.md`.

Файл должен быть понятен без технических знаний.

## Требования к operating-context.md

Опиши операционный контекст рутины:

- кто пользователь или команда;
- для чего нужен агент;
- какие источники используются и зачем;
- какие направления, типы клиентов, проектов или сущностей важны;
- какие действия агент может готовить;
- какие действия запрещены без согласия;
- какие действия требуют отдельного согласия;
- как работать с почтой;
- как работать с CRM;
- как работать с задачником;
- как работать с локальной базой;
- что делать при недостающих данных;
- правила языка и стиля.

Обязательно добавь безопасные правила:

- письма не отправлять автоматически;
- CRM не менять без точного согласия;
- stage/status изменения показывать отдельным действием;
- задачи не создавать без согласия;
- delete в задачнике только после отдельного подтверждения;
- секреты не хранить в markdown/YAML.

## Требования к routine-flow.md

Опиши пошаговый процесс регулярного прогона.

Включи:

- стандартный запуск;
- период по умолчанию;
- правило health-check источников;
- правило остановки при недоступном обязательном источнике;
- порядок сбора данных;
- классификацию апдейтов;
- универсальную action schema;
- список допустимых action types;
- правила проверки дублей;
- правила сборки HTML;
- финальный audit checklist.

Используй такую action schema:

action_id: string
client_or_contact: string
direction: string
system: CRM | Email | Task Manager | Calendar | Meeting Notes | local | other
action_type: string
proposed_action: string
why: string
source: string
duplicate_status: string
approval_status: string
exact_text: string

Используй такие action types:

- crm_note_create
- crm_note_append
- crm_stage_change
- email_draft_create
- email_draft_update
- task_create
- task_update
- handoff_prepare
- needs_missing_data
- no_action_duplicate

Одно действие должно быть одной карточкой. Не объединяй несколько клиентов, кандидатов или проектов в одну строку.

## Требования к data/*.yaml

Создай стартовые YAML-файлы с пустыми или примерными структурами.

`clients.yaml` или выбранный аналог:

- canonical name;
- aliases;
- domains или identifiers, если есть;
- CRM hints;
- notes.

`projects.yaml`:

- key;
- name;
- owner;
- kind;
- rules;
- task policy.

`materials.yaml`:

- direction или topic;
- canonical links;
- message snippets;
- rules for using materials.

`action-log.yaml`:

- applied_at;
- system;
- entity_type;
- entity_id;
- action;
- summary;
- source;
- applied_by.

Не добавляй реальные секреты, токены, пароли, OAuth-коды, refresh-токены, приватные ключи или содержимое `.env`.

## Требования к templates/review.html

Создай статический локальный HTML-шаблон для проверки регулярного прогона.

Он должен быть action-first:

1. Шапка с периодом, счетчиками и коротким статусом источников.
2. Decision strip: что проверить сначала, что готово к согласованию, что не менялось наружу.
3. `Вопросы / блокеры`.
4. `Готово к согласованию`.
5. `Детали`, только если они помогают принять решение.
6. `Уже применено / дубли`.
7. `Технический журнал`.

Требования к дизайну:

- локальный HTML без внешних запросов;
- русский язык;
- карточка 1 действие = 1 карточка;
- точные тексты писем, CRM notes, задач и handoff внутри `<details>`;
- блокеры визуально отличаются от действий;
- CRM, Email, Task Manager, handoff и duplicate имеют разные бейджи или цветовые акценты;
- радиус карточек не больше 8px;
- текст должен нормально помещаться на мобильном и desktop.

## Требования к runs/.gitkeep

Создай пустой файл, чтобы папка `runs/` сохранялась в проекте.

## Тестовый dry-run сценарий

В конце работы добавь в `START_HERE.md` пример запроса:

Запусти тестовый прогон за последние 24 часа. Проверь доступность источников и подготовь локальный HTML-файл проверки, но ничего не создавай и не записывай во внешние системы без моего согласия.

Не выполняй настоящий прогон и не подключайся к внешним системам, если я отдельно этого не попросил.

## Проверка результата

После создания файлов проверь:

- все файлы созданы;
- инструкции не привязаны к чужому продукту или чужой CRM;
- источники и правила взяты из моего брифа;
- есть правило остановки при недоступном обязательном источнике;
- есть запрет внешних изменений без согласия;
- stage/status изменения требуют отдельного согласия;
- есть action schema;
- есть проверка дублей;
- есть шаблон локального HTML;
- нигде нет просьбы вставлять секреты в markdown/YAML;
- первый запуск описан как dry-run.

Коротко сообщи, какие файлы созданы и как запустить первый тест.
```

## Как использовать этот промпт

1. Заполните бриф из `02-human-setup-guide.md`.
2. Скопируйте промпт выше в Codex или Claude Code.
3. Вставьте свой бриф в раздел `Мой бриф`.
4. Попросите агента создать файлы.
5. Проверьте результат.
6. Запустите первый dry-run без внешних записей.

Хороший первый запрос после сборки:

```text
Запусти тестовый прогон моей рутины за последние 24 часа. Если какой-то обязательный источник недоступен, остановись и скажи, что именно нельзя проверить. Ничего не создавай и не записывай во внешние системы без моего согласия.
```
